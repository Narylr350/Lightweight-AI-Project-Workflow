# Project Work

## 用途

日常主入口。根据用户需求和项目状态判断本轮任务类型，然后以最小范围推进。

不是单纯的写代码入口。必须先判断任务类型，再决定怎么做。

---

## 何时用

除非常简单维护改动外，每次开工的主入口。用户提出任何需求、问题、改动时，先经过这里判断。

---

## 何时不用

- 项目还没基线（无 `.ai/PROJECT.md` 或基线文件不完整）→ 先用 project-init
- 一轮工作刚结束，准备提交 → 用 project-finish
- 多轮失败需要诊断 → 用 project-audit
- 非常简单维护改动（如错别字、`.gitignore` 一行、格式小修）→ 直接做最小修改，不套完整流程

---

## 关键规则

### 开工前必读基线

1. **`.ai/PROJECT.md`** — 决策摘要。确认 Goal、MVP、Non-goals、Seed Tasks，检查当前需求是否与基线冲突。
2. **`.ai/CONSTRAINTS.md`** — 工作规则。确认硬约束、旧工作流权威源处理方式、已接入的执行层 skill。
3. **最近 commit 的 Next 字段** — 找到上一轮留下的交接线索。

按任务需要再读：

- **`.ai/TECH.md`** — 涉及技术方向、架构边界、依赖、平台选择时读。
- **`.ai/VALIDATION.md`** — 需要决定验证方式或进入 finish 前读。

### 读完后必须用 Git 核验

交接信息只是线索，可能过期或错误。开工前必须跑：

- `git status` — 当前工作区状态
- `git log --oneline -10` — 最近提交历史
- `git diff` — 未提交的改动

发现 commit message 说的和 Git 实际不一致时，以 Git 为准。不盲目按 Next 执行。

### 判断任务类型

读完基线和核验状态后，判断本轮属于哪种任务：

- **新功能** — 实现 MVP 或后续功能
- **修 bug** — 修复问题
- **复盘** — 梳理现状，不改代码（轻量版 audit）
- **黑盒观察** — 运行项目看行为，不改代码
- **文档同步** — 更新 `.ai/` 基线文件或其他文档
- **release 收尾** — 版本发布前的整理
- **提交前整理** — 整理工作区准备提交
- **需求变化** — 用户改变方向
- **继续上次任务** — 按 commit Next 继续。继续前必须先判断上次任务类型；继续不等于默认继续写代码。

### 快速模式：简单任务跳过深度探索

判断完任务类型后，如果任务是**范围明确、改动可预测、用户需求无歧义**的（配置改名、lint 修复、变量同步、单个文件文档更新），跳过深度项目探索（不用读 docs/context、docs/engineering、任务历史记录），直接做。

- 快速模式仍然读 `.ai/PROJECT.md` + `.ai/CONSTRAINTS.md` + git status/log/diff，这是安全和恢复锚点，不能省。
- 跳过的是"探索项目文档理解上下文"阶段，不是 git 核验。
- 只有任务范围模糊或涉及多个模块时才深度探索。让简单任务不因为"先读 5 个文档确认背景"而变慢。

判断完任务类型后，只做最小范围的改动。

- 能改一个文件解决，不改两个。
- 能加 10 行解决，不重构 100 行。
- 不顺手改无关代码。
- 不为"以后可能用到"加功能。
- 尊重原项目工程风格：代码风格、目录结构、命名、技术栈、测试命令、文档语言和语气要跟随项目。但本 pack 的工作流契约仍生效：`.ai/` 基线文件、结构化 commit、Git 核验、Decision Ownership 不被原项目习惯覆盖。
- 验证按风险分级：配置/文档/纯变量名等低风险改动用 targeted 验证（grep、单次命令），不用全量测试。不让验证成本超过改动成本。
- AI 不主动维护被项目 Non-goals 或 pack 工作流排除的文档。但用户明确要求维护的文档，不管是否被 gitignore 都做。

### 开发方法论与 skill 接入

开发方法论由用户选择（TDD / 自由编码 / 其他 skill），project-work 不强制。

- 如果 `.ai/CONSTRAINTS.md` 记录了已确认接入的执行层 skill（如 TDD / debugging / codebase-design），在 work 范围内使用。
- 不要默认强制 TDD，也不要禁止 TDD。
- 执行层 skill 只管"怎么做"，用完后回到 work 流程，不要被它接管整个任务流程。完成后按正常流程进入 project-finish。

### 任务拆解：按同类合并，不同类才拆

任务太大一轮做不完时，不要硬做，也不要逐文件拆。按**同类问题合并、不同类问题拆开**的原则拆：

- **同类问题一轮做完。** 判断标准：能不能用同一个验证方式一次性确认全部改对。能 → 同类，一起改、一次验证、一个 commit。不能 → 不同类，分开做。
- 例：3 个文件的 config 变量名漂移 → 同类，一轮做完。1 个 config 漂移 + 1 个路由鉴权策略 → 不同类，分开做。
- **不同类问题各自独立 commit。** 拆完后只执行第一个，其余写进 commit message 的 Next。
- 拆完后简短列出计划（一句话即可，如"本轮做 A/B，都属于 config 漂移；C 是鉴权策略，下一轮"）。不需要长文档，不需要逐条代码。
- 先做能跑通的最小路径，再补边界和细节。
- 平衡点：太粗容易乱和验证不充分，太细浪费 token。同类合并不丢质量，不同类拆开不丢聚焦。

### 分支规则

每轮工作开始前报告当前分支，并判断本轮和分支的关系：

1. **分支目标已完成** → 提示合并回主线。无风险（工作区干净、无冲突、验证通过）直接合并；有风险问用户。合并后回主线继续新任务。
2. **分支目标未完成，继续做** → 继续当前分支。
3. **当前需求与分支目标无关 + 简单维护** → 可以在当前分支顺手做。
4. **当前需求与分支目标无关 + 中等以上任务** → 切回主线或新开分支。
5. **在主线 + 存在未合并的任务分支** → 提示用户是否先合并。

分支创建策略：
- 简单维护改动可直接在 `main`/`master` 做（typo、文档小改、单行配置、低风险变量同步）。
- 中等以上任务建议使用任务分支：`feat/xxx`、`fix/xxx`。
- 高风险、多文件、多轮验证、可能失败或回滚的任务必须使用任务分支。
- 若当前已经在任务分支上，明确说明"继续当前任务分支"，不要重复建分支。

### 上下文压缩应对

AI 无法检测自己是否被压缩。压缩后丢失的信息无法自检恢复。

应对方式不是检测压缩，而是让关键约束不依赖对话记忆：

- 每轮 work 做完就 finish 提交，不留未提交的中间状态。
- work 开头固定读 `.ai/PROJECT.md` + `.ai/CONSTRAINTS.md` + git log + git status + 最近 commit Next，恢复约束和状态。
- 用户在对话中给出的重要约束（不在 `.ai/` 基线文件里的），work 应提示用户是否要写进基线。约束从对话记忆转移到文件后，压缩就丢不了。
- 已否方案、临时妥协等写进 commit message 的 Notes，不靠对话记忆。

### 验证环境处理

work 用到某环境时即时探测（端口探测、命令检测等轻量方式，几秒钟），不可用时按 `.ai/VALIDATION.md` 或 `.ai/CONSTRAINTS.md` 里记录的规则处理。

处理顺序：AI 先尝试修复（启动服务、设环境变量——可逆操作、不问用户）→ 尝试安装 → 确实做不到才提示用户。不把"环境没起"这种执行问题抛给用户决策。

如果 `.ai/` 基线文件没有对应规则，按优先级：不打断用户（AI 先试）> 降级验证方式（如浏览器验证不通过改查静态产物）> 记录阻塞跳过。

### 决策边界：已确认范围自动推进，基线变化停下

- 已确认范围内可以自动推进，不反复确认普通执行步骤。
- 一旦任务会改变目标、MVP、范围、技术方向、长期规则或 `.ai/` 基线，必须停下让用户确认。
- "继续"只能继续已确认的 Next 或当前任务边界，不能自动扩大成重构或新方向。

### 需求与基线冲突时停下

如果用户当前需求与 `.ai/PROJECT.md` 基线冲突（要做的事在非目标里，或超出 MVP 范围且方向不同）：

- **不要偷偷改基线**。
- **不要默默按新方向改代码**。
- 停下来问用户：这是临时探索，还是要更新基线？
- 如果是基线变化，建议更新对应的 `.ai/` 基线文件（或触发 project-init 重建）。
- 如果是临时探索，继续做，但提醒用户最后要在 commit message 的 Notes 里标明"偏离基线，待确认"。

如果连续多轮都偏离基线，触发 project-audit。

### 核心原则

详见 `docs/design-principles.md`。重点是：用户当前需求优先于旧计划，但不偷偷改基线；handoff 必须核验。

---

## 完成后下一步

有可提交改动且不是非常简单维护改动时，**进入 project-finish**：检查改动、验证结果、生成 commit message。

- 没有可提交改动 → 不需要 finish，直接结束。
- 发现多轮失败或偏航 → 建议触发 **project-audit**。
- 用户需求与基线严重冲突 → 建议触发 **project-audit** 或 **project-init** 重建基线。
- **当 commit Next 表达完成或无后续，且用户没有给出新需求时**，不自己编造新任务。停下来告诉用户当前状态，确认是要扩方向还是结束。

不要在 work 结束后留一个没有 Next 的空档。要么进 finish，要么明确告诉用户当前状态和下一步。
