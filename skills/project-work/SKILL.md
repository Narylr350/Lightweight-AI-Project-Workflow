## 用途

日常主入口。根据用户需求和项目状态判断本轮任务类型，然后以最小范围推进。

不是单纯的写代码入口。必须先判断任务类型，再决定怎么做。

---

## 何时用

每次开工的主入口。用户提出任何需求、问题、改动时，先经过这里判断。

---

## 何时不用

- 项目还没基线（无 PROJECT.md）→ 先用 project-init
- 一轮工作刚结束，准备提交 → 用 project-finish
- 多轮失败需要诊断 → 用 project-audit

---

## 关键规则

### 开工前必读两样东西

1. **PROJECT.md** — 确认目标、非目标、MVP。检查当前需求是否与基线冲突。
2. **最近 commit 的 Next 字段** — 找到上一轮留下的交接线索。

### 读完后必须用 Git 核验

交接信息只是线索，可能过期或错误。开工前必须跑：

- `git status` — 当前工作区状态
- `git log --oneline -10` — 最近提交历史
- `git diff` — 未提交的改动

发现 commit message 说的和 Git 实际不一致时，以 Git 为准。不盲目按 Next 执行。

### 判断任务类型

读完基线和核验状态后，判断本轮属于哪种任务：

- **新功能** — 实现 MVP 或后续功能
- **修 bug** — 修复问题
- **复盘** — 梳理现状，不改代码（轻量版 audit）
- **黑盒观察** — 运行项目看行为，不改代码
- **文档同步** — 更新 PROJECT.md 或其他文档
- **release 收尾** — 版本发布前的整理
- **提交前整理** — 整理工作区准备提交
- **需求变化** — 用户改变方向
- **继续上次任务** — 按 commit Next 继续

### 最小范围推进

判断完任务类型后，只做最小范围的改动。

- 能改一个文件解决，不改两个。
- 能加 10 行解决，不重构 100 行。
- 不顺手改无关代码。
- 不为"以后可能用到"加功能。

### 需求与基线冲突时停下

如果用户当前需求与 PROJECT.md 基线冲突（要做的事在非目标里，或超出 MVP 范围且方向不同）：

- **不要偷偷改基线**。
- **不要默默按新方向改代码**。
- 停下来问用户：这是临时探索，还是要更新基线？
- 如果是基线变化，建议更新 PROJECT.md（或触发 project-init 重建）。
- 如果是临时探索，继续做，但提醒用户最后要在 commit message 的 Notes 里标明"偏离基线，待确认"。

如果连续多轮都偏离基线，触发 project-audit。

### 核心原则

详见 `docs/design-principles.md`。重点是：用户当前需求优先于旧计划，但不偷偷改基线；handoff 必须核验。

---

## 完成后下一步

进入 project-finish：检查改动、验证结果、生成 commit message。
