# Project Finish

一轮工作结束时使用。检查、验证、生成结构化 commit message。

## 用途

一轮工作结束准备提交前，统一执行：

- 检查 git status / diff
- 检查验证结果
- 检查临时文件
- 判断 .ai/PROJECT.md 是否需要更新
- 生成结构化 commit message

## 何时用

- 一轮工作结束，准备提交前。

## 何时不用

- 还没动任何代码 / 文件——没东西可提交。
- 正在 project-audit 中——先完成审计。
- 非常简单维护改动（如错别字、`.gitignore` 一行、格式小修）——可直接用简短 commit message，不必生成完整 5 字段；前提是不改变基线、不影响行为、不需要交接 Notes。

## 关键规则

### 1. 检查清单

按顺序检查：

1. **git status**：确认改了哪些文件，是否有未预期的改动。
2. **git diff**：确认改动内容符合本轮意图。
3. **临时文件**：删除或归位临时文件（测试输出、调试文件等）。不要提交临时垃圾。
4. **验证结果**：确认本轮改动是否达到预期效果。手动验证或跑测试，记录验证方式。
5. **.ai/PROJECT.md**：只有基线变化（方向调整 / 技术边界变化 / MVP 重定义）时才更新。不记日常进度。

### 2. commit message 格式

生成 5 字段结构化 message：

```
<type>: <一句话摘要>

Why:    为什么做这个改动（动机 / 触发）
What:   具体改了什么（关键改动点，2-5 条）
Verify: 怎么验证的（命令 / 手动检查 / 测试结果）
Next:   下一步该做什么（线索，不是承诺）
Notes:  按下方规则决定是否写
```

`<type>` 用约定式提交词：`feat / fix / docs / refactor / chore / test`。

范例见 `docs/examples.md`。

### 3. Notes 字段规则

默认可省。

但本轮存在以下任一情况，Notes **必须写**：

- **已否方案**：被否决的实现路径，避免下一轮重走
- **临时妥协**：技术债 / 硬编码 / 跳过的边界
- **外部限制**：依赖版本锁 / 平台限制 / 环境差异
- **未验证风险**：猜测 / 未跑通的路径 / 待确认假设
- **用户明确改口**：与前次方向不同的决定
- **容易误导下一轮 AI 的信息**：反直觉的约定 / 隐藏状态

### 4. 不替用户提交

- 生成 commit message 后，展示给用户确认。
- 用户可修改 message。
- 提交动作由用户执行，或用户明确授权后执行。
- 不要自动 `git commit`。
- `tag` / `release` / `push` 必须用户明确确认，不能自动执行。
- 一轮提交流程的完成标志是用户确认已经提交，或用户明确授权 AI 提交且命令成功。只生成 commit message 不算完成。

### 5. .ai/PROJECT.md 更新与决策归属

- .ai/PROJECT.md 只有在用户确认基线变化，或存在 Git / 文件 / 历史证据支撑并经用户确认时才更新。
- commit message 的 Next 是线索，不是下一轮命令。下一轮 AI 必须用 Git 核验后再决定是否照做。
- AI 推荐和假设不是基线事实。

## 完成后下一步

用户确认提交 → 回到 project-work 继续下一轮，或结束本次工作。
