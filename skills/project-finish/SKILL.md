# Project Finish

一轮工作结束时使用。检查、验证、生成结构化 commit message。

## 用途

一轮工作结束准备提交前，统一执行：

- 检查 git status / diff
- 检查验证结果
- 检查临时文件
- 判断 `.ai/` 基线文件是否需要更新
- 生成结构化 commit message

## 何时用

- 一轮工作结束，准备提交前。

## 何时不用

- 还没动任何代码 / 文件——没东西可提交。
- 工作尚未完成，只是因为额度、上下文或工具中断必须换 AI——用户明确要求时用 project-handoff，不为交接提交半成品。
- 正在 project-audit 中——先完成审计。
- 非常简单维护改动（如错别字、`.gitignore` 一行、格式小修）——可直接用简短 commit message，不必生成完整 5 字段；前提是不改变基线、不影响行为、不需要交接 Notes。

## 关键规则

### 1. 检查清单

按顺序检查：

1. **git status**：确认改了哪些文件，分别核对 staged / unstaged / untracked，是否有未预期的改动。
2. **git diff**：确认改动内容符合本轮意图。
3. **临时文件**：删除或归位临时文件（测试输出、调试文件等）。临时输出、runtime、cache、logs、下载物、本地凭据等不应提交；必要时补 `.gitignore`。AI 工具自己的运行时目录（如 `.omo/`、`.codex/`、`.claude/`、`.cursor/` 等）也不应提交。`.ai/HANDOFF.local.md` 是 project-handoff 生成的本地临时接力材料，不能 stage 或提交；当前工作尚未完成时不要把它当垃圾提前删除。
4. **验证结果**：确认本轮改动是否达到预期效果。手动验证或跑测试，记录验证方式。
5. **Finish 层 skill**：检查 `.ai/CONSTRAINTS.md` 是否声明了 finish 层 skill。如有，在生成 commit message 前对当前 diff 调用。发现问题回 project-work 修复，不带着已知问题提交。没有声明或 `.ai/CONSTRAINTS.md` 不存在则跳过。
   - 如果声明的 finish 层 skill 当前环境不可用，不要静默跳过，也不要换成别的 skill。把它当作基线 / 环境不一致：提示用户选择临时跳过、修复环境，或更新 `.ai/CONSTRAINTS.md`。
   - Finish 层发现必须先分类：当前 diff 引入或暴露的阻塞问题（bug、死代码、行为不一致、验证缺口、安全/数据/权限/外部 I/O 风险）→ 回 project-work 修复，不生成提交 message；非本轮引入的既有问题或用户明确接受的非阻塞债务 → 可写 Next，Notes 按规则决定是否必写；误报 → 简短说明不采纳原因，必要时写 Notes 防止下一轮误解。
6. **基线文件**：只有基线变化（方向调整 / Tech Direction 变化 / Constraints 变化 / MVP 重定义 / Validation 规则变化）时才更新对应的 `.ai/` 文件。不记日常进度。
   - 如果当前在任务分支，且本轮要更新项目级 `.ai/` 基线文件，必须先确认这是分支专用基线，还是应回到 `main` / `master` 或合并分支后再做。
7. **gitignore / 文件归属检查**：只 stage 本轮确实应该进 Git 的文件；不要不看 status / diff 就 `git add .`。如果本轮改动涉及文档或其他交接材料，检查它们是否被 `.gitignore` 排除；project-init 生成的四个 `.ai/*.md` 基线默认属于应进 Git 的交接材料。如果被排除，在 Next 或 Notes 中说明这些文件不会进 Git，后续 AI 接手时可能看不到。

### 2. 触发式检查

以下检查不是每轮都做，只在 diff 触发条件出现时执行。检查范围只限本轮 diff 和被其直接影响的 Maintenance Surface，不扩大成全仓内容或项目气质审查。

**Maintenance Surface 检查** — 当本轮 diff 涉及以下任一情况时触发：

- 用户可见行为变化
- CLI 参数 / 命令示例
- 配置项
- 依赖
- 安装方式
- 公开 API
- 包元信息
- release 相关内容

触发后，检查 `.ai/CONSTRAINTS.md` 里记录的 Maintenance Surface 文件是否需要同步更新。不需要同步的跳过；需要同步的回 project-work 修复，或在本轮一并改完。

**版本号不在这处理。** 版本号只在 release / tag / publish / 用户明确要求时处理，不每轮 bump。

**产物内容边界检查** - 确认本轮 diff 中的代码、README、用户文档、产品文案只写项目内容。以下内容属于交接层（commit Notes）或基线层（`.ai/`），不属于产物层，发现则清理：

- 用户纠偏原话
- AI 失败过程
- AI 自我警告
- 内部设计讨论
- 当前开发状态、迁移方向、内部工作流规则（这些属于 `.ai/` 基线文件）

顶层 README（项目根目录）面向 GitHub 访客和项目使用者，写项目是什么、能干什么、怎么装、怎么用。子目录 README 的受众和用途跟随项目原有约定，不一刀切。改动 README 的结构或表达方式时，确认是否符合该 README 的受众定位。

### 3. commit message 格式

生成 5 字段结构化 message：

```
<type>: <一句话摘要>

Why:    为什么做这个改动（动机 / 触发）
What:   具体改了什么（关键改动点，2-5 条）
Verify: 怎么验证的（命令 / 手动检查 / 测试结果）
Next:   下一步该做什么（线索，不是承诺）
Notes:  按下方规则决定是否写
```

`<type>` 用约定式提交词：`feat / fix / docs / refactor / chore / test`。

commit message 正文（Why/What/Verify/Next/Notes 的内容）用用户的语言。用户说中文就写中文，不要默认套英文。字段名和 `<type>` 保持英文。

范例见 `docs/examples.md`。

### 4. Notes 字段规则

默认可省。

但本轮存在以下任一情况，Notes **必须写**：

- **已否方案**：被否决的实现路径，避免下一轮重走
- **临时妥协**：技术债 / 硬编码 / 跳过的边界
- **外部限制**：依赖版本锁 / 平台限制 / 环境差异
- **未验证风险**：猜测 / 未跑通的路径 / 待确认假设
- **用户明确改口**：与前次方向不同的决定
- **容易误导下一轮 AI 的信息**：反直觉的约定 / 隐藏状态

### 5. 不替用户提交

- 生成 commit message 后，展示给用户确认。
- 用户可修改 message。
- 提交动作由用户执行，或用户明确授权后执行。
- 不要自动 `git commit`。
- `tag` / `release` / `push` 必须用户明确确认，不能自动执行。
- 一轮提交流程的完成标志是用户确认已经提交，或用户明确授权 AI 提交且命令成功。只生成 commit message 不算完成。

### 6. 完成验证硬约束

- 没有验证就不能声称完成。Verify 字段不能写"未验证"或留空。
- 验证方式可以是：跑测试、手动操作确认、运行项目看结果。
- **验证结论不能超过证据覆盖范围**：构建通过只证明可以构建，单元测试只证明覆盖到的逻辑，逆向 / 规格证据只证明参考事实，不等于实现已经一致。视觉、交互、动画、性能、平台集成或真实外部 I/O 属于本轮验收目标时，必须用对应环境的手动或集成验证；未覆盖的范围不能写成整体完成。
- 如果确实无法验证（如纯文档改动），Verify 写明"手动检查：XXX 已确认"。
- **验证失败时不能进 finish，必须回到 project-work 修复。** 这条只适用于本轮实际执行了的验证。如果某验证因环境阻塞根本没执行（如 Docker daemon 没起、DB 没开），不视为"验证失败"——它不属于本轮验证范围。
- 环境阻塞的验证必须写进 Next（作为开放项，注明阻塞原因和解除条件）和 Notes（标记为未验证风险），不能在 Verify 里写"通过"或"跳过"假装验证过。

### 7. 基线文件更新与决策归属

- `.ai/` 下的基线文件（PROJECT.md / TECH.md / CONSTRAINTS.md / VALIDATION.md）只有在用户确认基线变化，或存在 Git / 文件 / 历史证据支撑并经用户确认时才更新。
- commit message 的 Next 是线索，不是下一轮命令。下一轮 AI 必须用 Git 核验后再决定是否照做。
- 写 Next 时，如果后续验证依赖 `.ai/VALIDATION.md` 中标记为不可靠的环境，应标明"可选验证，环境不可用时降级为 XXX"或给备选验证思路，不让下一轮 AI 在不可靠环境上死磕。
- AI 推荐和假设不是基线事实。

#### Next 字段诚实性规则

- 禁止用"基本完成""后续按需处理""阶段性完成"等话术掩盖已知未完成项。
- Next 只能写与当前 `.ai/` 基线一致、并由本轮 diff 或当前任务自然延伸出的最近下一步；不能把新方向、范围扩大、技术路线变化或长期规则变化写成默认下一步。
- 如果本轮有已知未完成的验证、未修复的阻塞、未跑通的路径，Next 必须诚实列出，并标注阻塞原因和解除条件。
- 如果 Notes 记录了可行动的未完成风险，该风险必须同时出现在 Next，不能让下一轮 AI 只在 Notes 里读到风险但在 Next 里看到"基本完成"。
- Next 可以短，但不能空。没有下一步就写 None。

### 8. 分支合并规则

提交成功后，如果当前在任务分支且满足以下条件，可以合并回主线：

- 工作区干净。
- 本轮验证通过。
- finish 层 skill（如有声明）检查通过或无阻塞发现。
- `main`/`master` 没有冲突或可快进。
- 没有未完成的阻塞项需要保留在分支里。
- 没有 tag/release/push 需求。

无风险时可以直接合并；有冲突、验证失败、未完成阻塞或会改变主线风险时，停下让用户确认。合并后仍不自动 push，push 必须用户明确确认。

## 完成后下一步

用户确认提交后：

- 如果存在 `.ai/HANDOFF.local.md`，且其中交接的工作已在本次提交中完成，删除该本地 handoff；如果无法确认是否已消费，保留并明确提醒，不擅自删除。
- 当前工作轮结束。即使处于目标模式、持续执行或已授权自动推进，执行下一项前也必须重新按 **project-work** 核验任务类型和分支关系。
- 有明确 Next → 建议用户在新对话用 **project-work** 继续。
- 多轮失败或偏航 → 建议触发 **project-audit**。
- 项目方向已变 → 建议触发 **project-init** 重建基线。
- 任务结束 → 结束，不要画蛇添足。

不要在 finish 结束后留空档。要么指向 work 继续，要么明确结束。
