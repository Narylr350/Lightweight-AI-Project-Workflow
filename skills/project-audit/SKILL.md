# Project Audit

低频复盘诊断工具。默认只分析，不改代码。

## 用途

诊断项目是否跑偏、交接信息是否与事实冲突、是否过度工程化。输出诊断报告和建议，交用户决定下一步。

## 何时用

- 多轮失败：连续几轮没解决问题。
- 用户喊停："不对" / "先复盘" / "怎么又这样"。
- release 前：发布前做一次整体检查。
- 新 AI 接手发现材料冲突：`.ai/` 基线文件 / commit message / 代码三者不一致。

## 何时不用

- 正常推进中——用 project-work。
- 只想提交——用 project-finish。
- 没有异常信号时不要常态化审计。

## 关键规则

### 1. 默认只读

- audit 结论是诊断，不是裁决。
- **不直接改代码**。
- **不改 `.ai/` 基线文件**。
- 输出诊断报告 + 建议，交用户决定。
- 发现基线过期 → 建议更新对应的 `.ai/` 基线文件，而非直接改。
- 发现 bug → 在报告里指出，交 project-work 修复。
- 要把 audit 结论固化成长期规则，必须用户确认或有可核验证据。AI 的诊断不能自动变成基线。
- 审计后不要自动修 skill。先区分通用设计缺口、规则冲突、单次输出偏差、外部工具问题；只有前两类才建议修改规则。
- 不要把非本项目的问题写进本项目的 commit Notes 或长期文档。

### 2. 检查 10 项

1. **是否跑偏**：最近若干 commit 做的事是否符合 `.ai/PROJECT.md` 的目标、MVP、Non-goals？是否触碰了非目标？
2. **文档是否过期**：`.ai/PROJECT.md` / `.ai/TECH.md` / `.ai/CONSTRAINTS.md` / `.ai/VALIDATION.md` 是否还和代码实际一致？
3. **handoff 是否和 Git 冲突**：commit message 的 Next / Notes 与 `git log` / `git diff` 实际状态是否一致？交接信息是否过期或错误？
4. **是否重复走已否方案**：是否有 commit 的 Notes 记录了已否方案，但后续又在重做？
5. **是否过度工程化**：是否引入了 MVP 不需要的复杂度（第三方库 / 抽象层 / 配置项）？
6. **流程规则是否跨入口覆盖**：如果失败和 workflow / skill 规则有关，检查是否只在某个入口写了规则，但其他入口仍能改同一类事实或绕过同一决策点。不要只修当前触发的 skill，要找出所有能触发同类动作的入口。
7. **维护表面是否与代码脱节**：README、CHANGELOG、版本文件、配置示例、API 文档、CLI help、安装说明等是否和最近代码改动一致？代码改了但维护表面没同步是常见缺口。
8. **产物是否混入非项目内容**：代码、README、用户文档、产品文案中是否混入了用户纠偏原话、AI 失败过程、AI 自我警告、内部设计讨论？这些属于交接层（commit Notes），不属于产物层。
9. **项目表达是否偏离已确认锚点**：只有项目级指令、`.ai/CONSTRAINTS.md` 或现有产物中存在明确表达锚点时才检查。使用通用模板、标准表达或平均做法本身不算问题；只有它覆盖了已确认的作者 / 项目选择，或未经确认改变了已有项目表达时，才算漂移。没有明确锚点时跳过，不从少量对话推导作者气质。
10. **代码质量**：最近改动的文件是否有已知代码质量问题（warning、error、过时 API、潜在 bug）。如果 `.ai/CONSTRAINTS.md` 声明了代码质量检查 skill，调用它；否则用 `.ai/VALIDATION.md` 里的验证命令做基本检查或跳过。

### 3. 输出格式

诊断报告包含：

- **触发原因**：为什么做这次审计。
- **现状（Git 核验）**：最近 commit / `.ai/` 基线 / 代码实际状态。
- **问题**：逐条列出发现的问题，对应 10 项检查。
- **根因**：为什么会跑偏。
- **建议**：具体的下一步行动（通常指向 project-work 的某个任务，或建议回 project-init 重建基线）。
- **Skill 效果评估**（条件触发）：如果本次对话使用了 `.ai/CONSTRAINTS.md` 声明的 skill，评估其效果：是否帮到任务、有无误报漏报、是否拖慢流程。给出建议：继续用 / 换 / 删 / 加新 skill。不自动改 CONSTRAINTS，交用户决定。

范例见 `docs/examples.md`。

### 4. 核验方式

所有结论必须基于 `git log` / `git status` / `git diff` 和文件实际内容。不要基于"印象"或"上次对话说的"下结论。

## 完成后下一步

把诊断报告交给用户决定。通常：

- 回到 project-work 做修正。
- 或回 project-init 重建基线（如果跑偏严重）。
